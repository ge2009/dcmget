# DcmGet 版本说明

## 未发布

- 统一工作台顶部显示可复制的局域网 HTTP 远程访问 URL，便于同一可信网络内的电脑直接打开管理页面；Profile 内部 Web 端口继续由系统自动分配且不向用户暴露。
- 统一工作台新增“总速度”，实时汇总所有正在运行 Profile 的接收速度；单个 Profile 暂停、停止或暂时不可达时按零计入，不影响其余实例展示。
- 新建任务固定默认不生成 PDI，任务页启用后可直接打开 PDI 根目录，完成后也可明确打开本次 PDI 结果目录；设置页继续管理 PDI 机构、目录、分卷和离线查看器参数。
- Profile 的内部 Web 地址改为系统自动维护的回环端点，不再向用户显示或要求配置 Web 端口；启动前自动避开保留端口、Profile 冲突和本机占用，统一工作台可通过 `/profiles/<编号>` 路径直达实例。
- 工作台视觉调整为 shadcn neutral/zinc 语义化层级，采用中性浅色背景、清晰边框和克制阴影，仅使用蓝色表达选择、焦点和进度；目录、下载、上传、刷新与设置图标采用离线 Motion 动效，并遵循系统“减少动态效果”设置。

## 3.7.0（2026-07-24）

- 工作台前端迁移为唯一的 Vite + React + TypeScript 实现，移除 NiceGUI 和旧静态页面入口；Python/FastAPI、DCMTK 下载核心、Profile 进程隔离、任务恢复、授权、PDI 和 Windows WebView2 外壳保持不变。
- 使用 `@base-ui/react`、Tailwind CSS、Motion、Zod 与 Lucide 建立离线设计和交互层；管理中心、Profile、任务、设置、日志、PDI、更新与运维继续使用原有同源 REST/SSE API、会话和 CSRF 边界。
- Node.js 仅参与开发与发布构建。成品前端固定输出为 `dcmget/webui-react/index.html`、`app.js`、`app.css` 和 `theme.js`，运行时不需要 Node.js、CDN 或外部字体与图标服务，并保持组件增量更新所需的稳定路径。
- 由于首次 React 版本会移除 NiceGUI 运行库和旧静态资源，3.6.1 升级到该版本需要安装一次完整 Windows 安装包；完成迁移后，固定文件名的前端更新可继续使用小体积组件增量包。
- 工作台首次打开默认使用专业医疗浅色主题，同时保留用户主动选择的深色偏好；统一放大正文、控件、日志和辅助信息字号，并同步调整导航栏、按钮、输入框和弹窗间距。
- 更新应用品牌图标为更清晰的 DICOM 切片接收标识，统一用于 Windows 程序、安装器、Web 工作台、favicon 与后续 macOS 资源。

## 3.6.1（2026-07-23）

- 自动更新改为只访问 `bwg-snell` 提供的 `https://dcmget.v2ex.com.cn/updates/` 固定通道，不再依赖 GitHub Releases；断网或专网环境只显示离线状态，不影响 Profile 与 DICOM 下载。
- 更新清单改为单文件 Ed25519 签名信封，客户端使用内置的独立更新公钥验证后，再检查版本、地址边界、大小、SHA-256、允许替换路径和安装树哈希；更新私钥与注册码私钥完全分离，不随客户端或更新站点发布。
- 小版本默认生成组件增量包，仅替换发生变化的应用文件；校验或替换失败时自动回滚并恢复 Windows 服务。Python、DCMTK、服务或安装布局变化时仍使用完整安装包，用户配置、Profile、恢复点、授权和影像数据不被覆盖。
- 这是新更新公钥的测试过渡版本。3.6.0 及更早安装需要手工安装一次 3.6.1 完整包；之后才能直接接收签名组件更新。同步精简管理页与下载页文案，移除口号式说明，保留操作提示和安全警告。

## 3.6.0（2026-07-23）

- NiceGUI 工作台增加 BaseUI 风格的设计变量层，统一 Quasar 按钮、输入框、卡片、弹窗、表格、通知与聚焦状态；保留专业医疗浅色主色和深色模式，不引入 React `baseui`、Node.js、CDN 或外部字体。
- Windows 统一管理中心增加软件更新面板：外网模式后台检查，发现新版后由服务器本机用户确认下载和安装；专网可关闭更新策略，关闭后不发起任何更新网络请求，断网检查也不阻断下载业务。
- 日常业务与 UI 更新优先使用指定基础版本的组件增量包，只替换 `DcmGet.exe` 和 `_internal/**` 中发生变化的应用文件；Windows 服务、DCMTK、依赖或安装布局变化时自动回退完整安装包。配置、Profile、任务恢复点、日志、授权、试用计数和下载影像不在更新覆盖范围内。
- 自动更新清单使用与已安装主程序同一签名者的 PKCS#7 签名，更新包再校验 SHA-256、文件大小、版本与 Authenticode。安装或组件替换由独立 SYSTEM 计划任务执行，失败时回滚已替换文件，避免服务停止时中断更新进程。
- Windows Release 工作流增加显式的正式发布与组件增量选项；只有完成 x64、Authenticode、时间戳和签名更新清单校验的构件才能进入 GitHub Releases 更新通道。
- 增加 `https://dcmget.v2ex.com.cn/updates/` 固定更新镜像和 `bwg-snell` 原子发布流程；镜像网络故障可安全回退 GitHub，签名或安全元数据异常仍保持拒绝更新。
- 3.5.x 及更旧版本不包含更新器，因此首次升级到 3.6.0 仍需手工运行一次完整安装包；后续版本才能使用管理中心的自动或增量更新。

## 3.5.2（2026-07-23）

- 修复自动重试中前一次已经归档有效 DICOM、最后一次却返回失败或无数据时，任务仍被错误显示为“失败/无数据”的问题；累计存在有效文件时现在保持“部分成功”，文件与恢复进度继续保留。
- DCMTK 原始 `E:/F:` 诊断继续写入详细日志，但不再永久占据工作台的红色错误摘要；自动重试最终仍失败时只显示一条可操作的综合错误，重试成功时不会残留误导性红色错误。
- PACS 建立关联超时现在明确显示目标 IP、端口以及“C-MOVE 尚未提交”，并以“本次未归档新文件”区分目标目录中的历史文件。
- `movescu` 异常断开且已建立关联时，归档前会短暂等待 `storescp` 子进程完成迟到文件写入，降低图像已经返回却因落盘竞态被误报为无文件的概率。
- 已保留有效图像但 PACS 未返回最终 C-MOVE 响应时，结果保持“部分成功”并安全暂停在当前检查号，不会将未确认收全的任务标为完成后继续下一条。

## 3.5.1（2026-07-22）

- 修复覆盖安装或升级时调用 WinSW 2.12.0 不支持的 `refresh` 命令而弹出 Windows 服务错误的问题；已有 `kayisoft-dcmget` 服务现在会在确认归属并停止后重新注册，继续保留升级前的启停状态、Profile 配置、授权与任务恢复数据。
- 修复工作台对当前 Profile 自有接收端口的误判：已有任务时先显示该任务正在使用端口，不再把自己的 `storescp` 当成外部占用；没有任务时仍会在预检中报告真实的外部端口冲突，启动接收器时再通过进程锁和监听检查复核。
- 当前 Profile 已有未完成任务时，工作台会直接说明端口由当前任务使用并禁用“开始下载”，引导用户继续、重试或结束当前任务，不再显示含义错误的端口冲突。
- 完成 Web 工作台的视觉收尾：实际使用的 NiceGUI 页面与备用静态页面均支持浅色/深色主题、跟随系统及本机持久化；补充分类型操作反馈、连接加载态、减少动态效果和 Windows 高对比度模式，并保持所有资源离线可用。

## 3.5.0（2026-07-22）

- Windows 本机入口改为独立的 WebView2 窗口，不再默认弹出系统浏览器；后台服务与窗口进程分离，关闭窗口不会停止正在执行的下载，重新打开快捷方式即可回到统一工作台。局域网电脑仍可通过浏览器访问管理地址。
- 将“开始下载”操作栏上移到页面主信息区下方，预检通过后无需滚动到底部；修复暂停过程中“继续”按钮文字与图标不可见、`pause_pending` 原始状态码直接显示的问题。
- 修改运行中 Profile 的配置时会先明确询问是否停止；确认后等待后台任务清理、Profile 锁释放及 Web/SCP 两个端口均可重新绑定，全部完成后才打开配置窗口。
- 修复停止接口忽略清理失败、服务监管与优雅停止竞态以及遗留 DCMTK 子进程仍占用端口的问题；无法确认归属的端口占用只报告阻塞原因，不会误杀其他程序。

## 3.4.1（2026-07-21）

- 修复 Windows 管理中心的 Profile 配置弹窗会随 4 秒状态刷新自动关闭的问题；配置弹窗改为挂载在稳定的页面容器中，刷新 Profile 卡片时不再销毁弹窗或丢失已输入内容。
- 优化 NiceGUI 工作台字体层级：中文界面优先使用 Windows/macOS 本机 UI 字体，技术数据与日志保留等宽字体；正文、表头和辅助信息提高到更清晰的字号，并避免非可变中文字体合成异常字重。

## 3.4.0（2026-07-21）

- Web 工作台改用 NiceGUI 重构，保留现有 FastAPI API、DCMTK 下载核心、Profile 进程隔离、任务恢复、授权与 PDI 流程；不引入 Node.js、CDN 或外部网络资源。
- Profile 主页收敛为“检查号 → 目标目录/PDI → 自动预检 → 开始/进度”主路径；运行中原位显示当前检查号、文件数、速度和可用操作，结果与错误就近展示。
- 超过 200 个检查号后仅保留聚合摘要，不渲染明细列表；日志默认只显示错误，用户主动开启后才显示详细日志。
- 设置按 PACS 服务端、本地 DICOM 接收端、存储/PDI 与高级选项分组；端口继续使用纯文本输入，避免滚轮误改。Windows 管理中心使用紧凑 Profile 卡片完成创建、启动和停止，并在管理端口内原位进入工作台、返回管理页或快速切换 Profile。
- NiceGUI 挂载在 `/workspace/`，所有业务变更仍通过原有同源 API 与 CSRF 边界；Socket.IO 通道新增 Host、Origin、IP 绑定会话与管理端私网来源校验，NiceGUI 所需内联 CSP 仅在工作台路径放行。
- Windows x64 构建开始收集 NiceGUI 的 Python 模块与离线 Vue/Quasar/Socket.IO 资源；安装版、便携版与 ZIP 仍只支持 x64 Windows，并允许 Windows ARM64 通过 x64 兼容层运行。

## 3.3.0（2026-07-20）

- Windows 管理中心升级为固定使用 `8786` 的统一单页工作台：左侧快速切换 Profile，右侧原位显示当前 Profile 的下载任务、进度、PDI、错误日志与运行摘要，不再跳转到不同端口或打开多个窗口。
- Web 界面采用本地实现的 BaseUI 风格设计系统，统一状态标签、按钮、卡片、空状态和设置抽屉；继续使用原生 HTML/CSS/JavaScript，不引入 React、Node.js 运行时、CDN 或外部网络依赖。
- 每个 Profile 仍由独立 Python/FastAPI 进程承载。管理中心通过只允许本机回环目标和固定 API 白名单的同源代理访问任务、配置与事件，Profile 的 Web 端口不再直接暴露为日常操作入口。
- 新建 Profile 默认停止；用户明确启动或停止后，期望运行状态独立保存在管理目录。Windows 或 `kayisoft-dcmget` 服务重启后只恢复用户选择运行的 Profile，未选择的 Profile 不会自动启动。
- 首次安装会创建可配置但保持停止的默认 Profile；Profile 配置、任务恢复点、授权、下载结果和 PDI 设置继续按原位置保存，升级不覆盖现有数据。
- Windows 开始菜单的服务快捷方式改名为“启动后台服务”和“停止后台服务”，主快捷方式直接打开统一工作台。本版本继续只生成 Windows x64 发布物。

## 3.2.0（2026-07-20）

- Windows 安装版新增独立的 DcmGet 管理中心，固定使用 `8786`；Profile Web 端口继续从 `8787` 开始。单个 Profile 的 AE、SCP 或 Web 端口配置错误不再导致统一管理入口一起离线。
- `kayisoft-dcmget` 同时守护管理中心和全部已启动 Profile；新 Profile 会在完成配置并首次启动后动态纳入崩溃重启监管。服务停止、升级或卸载时统一清理管理中心、Profile、`storescp`、`movescu` 与 PDI 子进程树。
- Profile 管理改为紧凑卡片工作台，统一显示 PACS、调用/接收 AE、SCP/Web 端口、运行状态和冲突原因，并可跳转到每个 Profile 的任务、设置与管理页面。
- 远程管理链接自动沿用当前 Windows 主机地址，不再误跳到访问者自己的 `127.0.0.1`；修改操作仅接受回环或可信私有局域网的实际 TCP 客户端，并继续启用 Host、Origin、会话与 CSRF 校验。
- Windows 桌面和开始菜单主入口改为管理中心；本版本仅发布 Windows x64 安装版、便携版和独立 ZIP。

## 3.1.1（2026-07-20）

- Web 工作台升级为更紧凑的临床操作界面：内置统一 SVG 图标、明确的导入与启动步骤、新建任务区可收起，并以克制的页面、连接和进度动效强化状态反馈；系统启用“减少动态效果”时自动关闭动效。
- 1024×768 窄桌面继续保持任务双栏，预检通过后压缩成功详情，使“开始下载”与修复提示保持在首屏；小于 840px 才切换为单栏与底部导航。
- 修复检查号输入防抖与启动确认之间的竞态；任务开始前会冻结并复核已经预检的检查号、Profile、PACS、保存目录和 PDI 设置，确认期间内容变化时拒绝提交。
- 目录选择只接受最新成功响应，高频任务与 PDI 操作增加互斥锁；补齐 PDI 校验终态、Profile 表单原生校验、“接受已有文件”确认，并避免实时日志更新抢走页面滚动位置。

## 3.1.0（2026-07-20）

- Windows x64 安装版新增自动启动的 `kayisoft-dcmget` 系统服务，统一启动和停止全部既有 Profile 及其 `storescp`、`movescu`、PDI 子进程；开始菜单和 Web 运维页均提供一键启停入口。
- 安装、覆盖升级和卸载会先停止现有服务，再按安装目录精确清理 DcmGet 进程树；不会误杀其他目录中同名的 DCMTK 工具，用户配置、恢复点、注册码和下载结果继续保留。
- 移除首次密码设置、登录、开始任务和关闭后台时的应用密码验证；本地维护与服务启停无需应用密码，裸 HTTP 仍仅限受信任医院内网使用。
- Profile 管理支持在启动前编辑 PACS、AE、SCP/Web 端口和目标目录；同一 Profile 的 SCP/Web 端口、不同 Profile 之间的端口以及系统占用都会在启动前给出明确提示。
- Profile 快捷方式改为直接打开对应的本地 Web URL；Windows 使用 `.url`，macOS 使用 `.webloc`，Linux 使用 URL desktop 文件。

## 3.0.1（2026-07-20）

- 开始下载和在服务器本机关闭 DcmGet 后台前，必须再次输入当前管理员密码；关闭浏览器标签页仍不会停止后台下载。
- 修复实际收到 DICOM 后，验收台账读取 `ReceivedDicomMetadata.attribution_status` 导致记录失败的问题；缺少显式归属状态时继续按检查号、Study UID 和元数据计算，不改变下载结果。
- PDI 新安装默认关闭，已由用户明确保存的设置仍保留；工作台初始化和任务草稿变化后会自动执行启动预检，仍可手动重新检查。
- 设置页将 PACS 服务端与本地 DICOM 接收端分成独立配置组，避免 PACS AE、本机调用 AE 和接收 AE 混在一起。

## 3.0.0（2026-07-20）

- 桌面界面由 PyQt5 全面替换为随程序离线内置的中文 Web 工作台；启动后使用系统浏览器访问，浏览器关闭不影响后台下载，运行时不依赖 Node.js、CDN、WebView2 或外部网络。
- 3.0.0 起客户端运行时不再包含或依赖 PyQt5；历史版本中的 Qt 说明只代表对应旧版本。
- Web 服务默认以裸 HTTP 监听 `0.0.0.0:8787`，支持可信局域网其他电脑访问。页面持续显示未加密提示；登录、密码哈希、HttpOnly 会话、CSRF、Host/Origin 校验和登录限流仍保留，不提供公网暴露模式。
- 启动时不再同步执行 DNS/FQDN 查询；即使医院内网 DNS 或 mDNS 不可用，也会直接使用本机主机名和网卡 IP 启动 Web 服务，避免界面长时间无响应。
- 保留一个 Profile 一个独立后台进程的运行模型；每个 Profile 的配置、恢复点、日志、DICOM 接收端口和 Web 端口互相隔离，新建 Profile 会自动避让已配置端口。
- 新工作台集中提供检查号导入、预检、进度与速度、暂停/继续/停止、失败重试、PDI、配置、授权、日志和运维工具；超过 200 个检查号继续只返回聚合进度，默认仅显示错误日志。
- 配置升级到 v8，新增 Web 监听地址、端口、自动打开浏览器和会话有效期；旧 PACS、AE、目标目录、匿名化、PDI、授权与任务恢复数据原样保留。
- Windows x64 安装版继续包含精简 DCMTK 与离线 OHIF，并为 `storescp.exe` 和 DcmGet Web 服务分别创建仅限域/专用网络的入站规则；仍拒绝 32 位运行时，Windows ARM64 通过 x64 兼容层运行。
